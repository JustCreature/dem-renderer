Interesting — 1.4 GB/s vs 0.6 GB/s. 2.3× faster, even though SIMD "shouldn't help" random access. Let's understand why.                                        
                                                                                                                                                                 
  ---                                                                                                                                                            
  The improvement is not from SIMD arithmetic                                                                                                                    
                                                                                                                                                                 
  The vaddq_f32 at the end is irrelevant. What actually helped is that you're now issuing 16 independent loads per loop iteration instead of 1.                  
                                                                                                                                                                 
  Scalar loop:                                                                                                                                                   
  iteration: load data[indices[i]]   → 1 cache miss in flight                                                                                                    
             wait...wait...wait (100 ns)                                                                                                                         
             fadd                                                                                                                                                
             next iteration                                                                                                                                      
                                                                                                                                                                 
  Your SIMD loop:                                                                                                                                                
  iteration: load data[indices[0..15]]  → 16 independent loads issued                                                                                            
             out-of-order engine overlaps all 16 simultaneously                                                                                                  
             more misses in-flight → better DRAM utilization                                                                                                     
                                                                                                                                                                 
  The CPU's out-of-order engine has a limited number of load buffer slots — maybe 100+ on M4. By issuing 16 loads before waiting, you fill more of those slots,  
  overlapping more DRAM round-trips simultaneously. This is memory-level parallelism (MLP), not compute parallelism.                                             
                                                                                                                                                                 
  ---                                                                                                                                                            
  Why not 16× improvement then                                                                                                                                   
                                                                                                                                                                 
  Two reasons:                                                                                                                                                   
                                                                                                                                                                 
  1. The temporary stack arrays in your implementation add overhead — each gather is: random DRAM load → store to stack → NEON load from stack. The stack stores 
  are cheap (L1) but they're extra work.                                                                                                                         
  2. The out-of-order engine was already overlapping some misses in the scalar version (~10-15 in-flight). You increased that, but the load buffer has a ceiling.
                                                                                                                                                                 
  ---                                                                                                                                                            
  The complete picture                                                                                                                                           
  ┌────────────────────┬───────────┬──────────────────────────────────┐                                                                                          
  │      Pattern       │ Bandwidth │            Bottleneck            │                                                                                          
  ├────────────────────┼───────────┼──────────────────────────────────┤                                                                                          
  │ seq_read scalar    │ 6.7 GB/s  │ compute (scalar loop)            │                                                                                          
  ├────────────────────┼───────────┼──────────────────────────────────┤                                                                                          
  │ seq_read SIMD      │ 21.8 GB/s │ memory bandwidth (single stream) │                                                                                          
  ├────────────────────┼───────────┼──────────────────────────────────┤                                                                                          
  │ random_read scalar │ 0.6 GB/s  │ DRAM latency (low MLP)           │                                                                                          
  ├────────────────────┼───────────┼──────────────────────────────────┤                                                                                          
  │ random_read SIMD   │ 1.4 GB/s  │ DRAM latency (higher MLP)        │                                                                                          
  └────────────────────┴───────────┴──────────────────────────────────┘                                                                                          
  The fundamental insight: different bottlenecks respond differently to SIMD. Sequential was compute-bound — SIMD crushed it. Random was latency-bound — SIMD    
  helped only indirectly through MLP. No optimization can make 0.6 GB/s match 21.8 GB/s because the problem is the access pattern, not the instruction choice.   
                                                                                                                                                                 
  This is exactly why Phase 1 focuses on memory layout before touching SIMD again. Ready to move on?
