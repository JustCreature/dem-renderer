# Phase 0 — Foundations & Tooling Setup

## Task 1: Cargo Workspace

A Cargo workspace is a collection of crates sharing one `target/` directory and one lock file. The workspace `Cargo.toml` just declares members — it has no `[lib]` or `[bin]` itself (except yours has `src/main.rs` as the root binary).

The structure from your plan:

```
dem_renderer/
├── Cargo.toml          # [workspace] manifest
├── src/main.rs         # binary crate (implicit, belongs to root)
└── crates/
    ├── dem_io/
    │   ├── Cargo.toml
    │   └── src/lib.rs
    ├── terrain/
    ├── render_cpu/
    ├── render_gpu/
    └── profiling/
```

The root `Cargo.toml` needs two things:
```toml
[workspace]
members = ["crates/dem_io", "crates/terrain", ...]
resolver = "2"  # use feature resolver v2

[package]
name = "dem_renderer"
# ...
```

Each crate under `crates/` is a standard `[package]` + `[lib]` Cargo.toml. The dependency chain flows upward — `dem_io` doesn't depend on `terrain`, `terrain` depends on `dem_io`, etc. Declare that with a path dependency:

```toml
# terrain/Cargo.toml
[dependencies]
dem_io = { path = "../dem_io" }
```

**Your task**: `cargo new --lib crates/dem_io` and repeat for each crate. Then wire the workspace root. You don't need to write any logic yet — just `pub fn placeholder() {}` in each `lib.rs` and verify `cargo build --workspace` compiles clean.

## Task 2: Profiling Harness (`profiling` crate)

The key hardware concept: a CPU cycle counter is a hardware register that increments every clock cycle. On x86-64 it's the TSC (`Time Stamp Counter`), read with `rdtsc`. On AArch64 (Apple Silicon) it's `cntvct_el0`, a virtual timer counter — note it ticks at a *fixed* frequency (usually 24 MHz on Apple Silicon), not the CPU clock frequency, so you need to convert to cycles using the current CPU frequency if you want cycles. That's an important gotcha.

The Rust wrappers live in `core::arch`:

```rust
// x86-64
unsafe { core::arch::x86_64::_rdtsc() }  // returns u64

// AArch64
unsafe { core::arch::aarch64::__cntvct_el0() }  // returns u64 (timer ticks, not cycles)
```

For a timing macro, the pattern is:

```rust
let t0 = read_counter();
black_box(your_work());  // std::hint::black_box prevents dead-code elimination
let t1 = read_counter();
let elapsed = t1 - t0;
```

`std::hint::black_box` is critical — without it, the compiler may optimize away the work you're trying to measure because the result isn't used.

**Your task**: implement two functions in `profiling/src/lib.rs` — `now() -> u64` (dispatches to the right arch) and a `timed<F: Fn()>(label: &str, f: F)` function that prints `label, elapsed` to stdout in CSV form. Annotate `now()` with `#[inline(never)]` during development (so it shows as a distinct symbol in profiler output), then switch to `#[inline(always)]` for actual measurements to minimize measurement overhead.

## Task 3: Baseline Memory Bandwidth

This is the most important calibration exercise in the whole project. The goal is to empirically discover your machine's memory bandwidth ceiling, which will bound every phase that follows.

**Why sequential vs random?**

Sequential access trains the hardware prefetcher — it detects a stride-1 access pattern and issues prefetch requests before you need the data. Random access defeats this: each cache miss is a full DRAM round-trip (~60–80 ns, ~200+ cycles). The ratio between the two numbers (should be ~20–40×) tells you exactly how expensive a cache miss is on your specific hardware.

**The experiment design:**

Allocate 64 MB of `f32` (= 16M elements). Then:

1. **Sequential read**: iterate `0..len` in order, sum all values, `black_box` the sum.
2. **Random read**: generate a random permutation of indices, follow them, sum values.
3. **Sequential write**: fill array in order.
4. **Random write**: write to random indices.

Time each with your profiling harness (or Criterion), report in GB/s = `(bytes_accessed) / (elapsed_seconds)`.

**What numbers to expect on Apple Silicon (your platform):**
- Sequential: ~70–100 GB/s (unified memory bandwidth)
- Random: ~2–5 GB/s (DRAM latency bound, even with 128 KB L1 helping a little)

For the random access to actually be random (not cached), the array must be large enough to not fit in L3. On M-series, L3 varies — to be sure, use 256 MB instead of 64 MB if you want to escape L3 entirely.

**Important**: the random index generation itself must not be in the hot loop. Pre-generate the permutation, then in the timed region just follow indices.

```rust
// Skeleton to guide your thinking:
let data: Vec<f32> = (0..N).map(|i| i as f32).collect();
let indices: Vec<usize> = generate_random_permutation(N);  // pre-compute

let t0 = profiling::now();
let mut sum = 0.0f32;
for &idx in &indices {
    sum += data[idx];
}
std::hint::black_box(sum);
let t1 = profiling::now();
```
