Key finding: GPU rendering is 8–63× faster than CPU on every system. The GTX 1650 achieves the largest ratio (63×) because the i5-9300H CPU is weak relative to the discrete GPU. M4's unified memory architecture gives the highest absolute fps (46.4) despite not having a discrete card.

Phase 6 Experiments

Exp 1: Tile Size Sweep

~7–10× tiling penalty on every system. Tile size (T=32 to T=256) makes no difference — the bottleneck is a continue branch inside the tiled loop that blocks auto-vectorisation. The cache hierarchy is irrelevant when the vectoriser is blocked.

Exp 4: AoS vs SoA

SoA advantage scales with bandwidth pressure: M4 1.0×, i7 1.8×, Win 2.3×, Asus 2.7× (nz-only parallel). On M4 the kernel is compute-bound; bandwidth has headroom so layout doesn't matter. On bandwidth-starved systems (Asus), SoA's 3× reduced cache traffic is decisive. (AoS = opposite layout)

Exp 9: TLB Working Set Sweep

M4 TLB knee at 4 MB (L1 DTLB exhaustion: 256 entries × 16 KB pages). x86 systems see their knee at ~256 KB–1 MB (4 KB pages, fewer entries). The 26 MB heightmap fits M4's L2 TLB (48 MB) but overflows typical x86 L2 TLBs, causing page-walk overhead during raymarching on Intel.

Exp 7: NEON Accumulator Experiment (M4 Mac only)

Breaking the serial dependency chain gives 3.2× speedup (1 → 4 accumulators). vfmaq has 4–5 cycle latency; a single accumulator creates a 4-cycle dep chain per iteration. Four independent chains fill all pipeline slots. Beyond 4 accumulators the kernel becomes memory-bound.

Cross-Cutting Conclusions

1. Vectorisation gates everything — the universal lesson

The Phase 6 tile size sweep showed the same ~7–10× penalty on every system from M4 to Asus. Root cause: a continue branch inside the tiled loop blocks auto-vectorisation. Cache topology, page size, and SIMD width become irrelevant when the vectoriser can't fire. Fix the vectorisation barrier first.

2. PCIe readback is the GPU fps ceiling on discrete systems

Win GTX 1650: ~20 ms GPU compute + ~47 ms PCIe readback = 67 ms/frame (15 fps). Eliminating readback via wgpu Surface/swapchain → ~50 fps. This is not unique to this project — any GPU compute loop with CPU readback is PCIe-limited when the GPU is fast enough.

3. Unified memory (M4) eliminates the PCIe tax

M4 Mac achieves 46 fps GPU despite not having a discrete card — because there is no PCIe to cross. GPU and CPU share the same 400 GB/s bus. Readback is fast enough that it doesn't dominate. On x86 + discrete GPU, the architectural split is the dominant bottleneck.

4. GPU beats CPU for parallel work; CPU beats GPU for serial work

Raymarching (embarrassingly parallel): GPU wins 8–63×. Shadow sweep (serial running-max per row): CPU NEON wins on every system. The GPU can't pipeline across the serial dependency chain, and shader cores at lower clock without branch prediction advantage don't beat a well-vectorised CPU loop.

5. Write-path bandwidth is 2–10× narrower than read-path

Write/read BW ratios: M4 0.40, Mac i7 0.23, Win i5 0.09, Asus 0.52. Store-buffer congestion + write-allocate RFOs cause this. Write-heavy kernels (stencil with output) saturate earlier than read-only kernels. Design hot inner loops to minimise writes.

6. SoA advantage scales with bandwidth starvation

SoA gain for nz-only parallel: M4 1.0×, Mac i7 1.8×, Win 2.3×, Asus 2.7×. On M4 the kernel is compute-bound, layout is irrelevant. On weak systems with narrow DRAM, AoS wastes 2/3 of every cache line and halves effective throughput. Target data layouts for the most constrained hardware.

7. M4's 16 KB pages give 4× more TLB coverage than x86

M4 L1 DTLB covers ~4 MB (256 entries × 16 KB), x86 covers ~1 MB (256 entries × 4 KB). The 26 MB heightmap fits M4's L2 TLB (48 MB) but overflows x86 TLBs. This reduces page-walk frequency during raymarching, contributing to M4's per-ray throughput advantage beyond just clock speed.

Glossary

TLB — Translation Lookaside Buffer. A small hardware cache (~64–3000 entries) storing recent virtual→physical address translations. TLB hit = 1 cycle. TLB miss = full page-table walk = 50–200 extra cycles.
RFO — Read For Ownership. Before writing to a cache line not in L1, the CPU fetches the full 64-byte line first. Every cold write causes a read — doubling DRAM write traffic.
LLC — Last Level Cache. The cache level closest to DRAM, shared by all cores. L3 on x86 (8–32 MB). SLC on M4 (32+ MB), shared with GPU and Neural Engine.
PCIe — Peripheral Component Interconnect Express. Bus connecting CPU to discrete GPU. Gen3 ×16 ≈ 16 GB/s each direction. Reading 3.4 MB pixels back to CPU costs ~47 ms in practice.
SIMD — Single Instruction Multiple Data. One instruction operates on multiple elements simultaneously. ARM NEON = 4×f32. x86 AVX2 = 8×f32 per instruction.
NEON — ARM's 128-bit SIMD instruction set. 4×f32, 4×i32, 8×i16, or 16×i8 per instruction. Used for vmaxq_f32 (shadow running-max), vfmaq_f32 (dot products).
AVX2 — Advanced Vector Extensions 2. x86 256-bit SIMD — 8×f32 per instruction. Available on Win GTX (i5-9300H) and Mac i7; NOT on Asus Pentium N3700.
SoA — Struct of Arrays. Each field is a separate array: [nx,nx,...][ny,ny,...][nz,nz,...]. Reading only nz loads 1/3 the cache lines vs AoS.
AoS — Array of Structs. Fields interleaved per element: [{nx,ny,nz}, ...]. Reading only nz wastes 2/3 of every cache line on nx and ny.
stencil — Computation where output[i] depends on a fixed neighbourhood around i. Your normal map is a stencil: reads pixels at ±1 row/col. Your Phase 6 benchmark uses a 5-point stencil (N/S/E/W/centre).
cardinal direction — North, south, east, or west exactly. Cardinal sun = shadow rays travel along one row or column = sequential memory access. Diagonal sun = rays cross rows = strided = cache misses every step.
DDA — Digital Differential Analyzer. Integer stepping algorithm that traces a line on a grid one pixel at a time. Used for shadow rays: step in the sun direction, compare heights at each grid cell.
ROB — Reorder Buffer. Tracks all in-flight instructions in an out-of-order CPU. Size determines how far ahead the CPU looks for independent work. M4 ≈ 600 instructions; typical x86 ≈ 350–500.
store buffer — Small hardware buffer (~50–80 entries) holding pending store instructions. Lets the pipeline continue after a store without waiting for cache write completion. When full → pipeline stalls.
write-allocate — Cache policy: on a write miss, fetch the full cache line before writing. Necessary because you're only modifying part of a line. Causes every cold write to also generate an RFO (read).
SoC — System on a Chip. CPU, GPU, memory controller, I/O all on one die, connected by on-chip fabric. M4 Max is a SoC — GPU and CPU share the same 400 GB/s memory bus, no PCIe between them.
PGD / PUD / PMD / PTE — The 4 levels of the x86-64 page table. PGD = root (address in CR3). PUD / PMD = middle levels. PTE = leaf containing the physical page address. Walking all 4 = 4 potential DRAM reads.
L1D / L2 / L3 — Cache levels. L1D ≈ 32–128 KB, ~4 cycles, private per core. L2 ≈ 256 KB–4 MB, ~12 cycles, private. L3 ≈ 8–32 MB, ~30–50 cycles, shared. Beyond L3 is DRAM (~80 ns = ~300 cycles on M4).
GpuScene — Design pattern: all GPU resources (heightmap, normals, bind groups) allocated once and reused. Only 128 bytes of camera uniforms written per frame. Eliminates per-frame re-upload overhead.
DDR4 — Double Data Rate 4. Common laptop/desktop RAM 2017–2022. Typical dual-channel bandwidth ~50 GB/s. Replaced by DDR5 in newer systems. Win GTX and Asus run on DDR4/DDR3L.
wgpu — Rust GPU API. Cross-platform over Vulkan/Metal/DX12. Used for all compute shaders. wgpu Surface = swap chain for presenting frames directly to the screen without CPU readback.
stack vs heap — Stack: compiler-managed, automatic lifetime, fixed size known at compile time, ~1–8 MB total. Heap: programmer-managed (Vec, Box), flexible size, persists across function calls, can use all available RAM.
